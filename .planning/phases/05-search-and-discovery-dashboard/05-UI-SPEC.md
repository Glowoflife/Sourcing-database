# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | shadcn |
| Preset | `base-nova` |
| Component library | base-ui-backed shadcn primitives |
| Icon library | `lucide-react` |
| Font | Geist via `next/font/google` (`--font-sans`) |

**Detected baseline:** `components.json` confirms `base-nova`, Tailwind v4, RSC enabled, and official shadcn registry only. Existing product language is already defined by zinc-neutral surfaces, compact borders, 10px radius (`--radius`), and simple sidebar navigation.

**Phase 5 extension rule:** Keep the existing internal-dashboard language. Do not introduce a new brand, hero treatment, display graphics, gradient-heavy backgrounds, or marketing-style cards. Add only a restrained industrial blue accent system for discovery-specific emphasis.

**Phase 5 component set:** `table`, `button`, `badge`, `input`, `popover`, `command`, `separator`, `sheet`, `skeleton`, `scroll-area`.

## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Inline icon gaps, badge padding adjustments, query highlight inset |
| sm | 8px | Tight control gaps, stacked metadata spacing, compact card padding |
| md | 16px | Default control spacing, table cell inner layout, section spacing |
| lg | 24px | Filter bar padding, card padding, page section gaps |
| xl | 32px | Desktop column gaps, section breaks, detail page block spacing |
| 2xl | 48px | Major page separation between title area and data region |
| 3xl | 64px | Reserved for full-page empty/error states only |

Exceptions:
- Mobile filter actions and tap targets must be at least 44px high.
- Results table rows should target a 64px visual rhythm with two-line cells where needed.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px | 400 | 1.5 |
| Label | 12px | 400 | 1.4 |
| Heading | 20px | 600 | 1.2 |
| Display | 28px | 600 | 1.15 |

**Rules:**
- Use only two weights in Phase 5: `400` and `600`.
- `Display` is reserved for page titles and manufacturer names on the detail header.
- `Heading` is for section titles such as `Products & CAS`, `Locations`, and `Contacts`.
- `Label` is for table headers, filter labels, metadata labels, and result counts.
- Monospace is allowed only for CAS numbers and raw extracted capacity text snippets.

## Color

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `background` / white + zinc text | Page canvas, table canvas, detail page body |
| Secondary (30%) | `card`, `sidebar`, `zinc-50`, `zinc-100` | Sidebar, filter bar, section cards, sticky summary rail |
| Accent (10%) | `blue-600` with `blue-50` and `blue-100` tints | Search emphasis and active discovery states only |
| Destructive | `destructive` / red | Error surfaces, destructive copy only |

Accent reserved for:
- Active `Manufacturers` nav item treatment
- Applied filter chips and active filter count
- Search input focus ring and explicit search submit state
- Manufacturer name links and exact-match query highlights
- Current pagination state and selected sort direction indicator

**Do not use accent for:**
- Every button
- Passive metadata pills
- Full-page backgrounds
- Success or warning states

**Status semantic palette:**

| Status | Background | Text | Usage |
|--------|------------|------|-------|
| New | `blue-100` | `blue-700` | Lead exists but not yet processed |
| Processing | `amber-100` | `amber-800` | Active crawl/extract work |
| Crawled | `emerald-100` | `emerald-700` | Content acquired, extraction pending |
| Extracted | `teal-100` | `teal-700` | Technical profile available |
| Errored | `red-100` | `red-700` | Crawl or extraction failed |

**Contrast rule:** All muted secondary text must still meet readable contrast against white or zinc-50 surfaces. Use `text-zinc-600` or darker for persistent labels.

## Visual Hierarchy

### Discovery screen focal points

1. The sticky search/filter toolbar is the primary focal point.
2. The results table is the dominant body element and must visually outweigh surrounding chrome.
3. The result count and active-filter summary are secondary cues placed directly below the toolbar.

### Detail screen focal points

1. The manufacturer name plus status header is the primary focal point.
2. The `Products & CAS` section is the first heavy data block in the main column.
3. The sticky right rail provides supporting context only; it must not visually dominate the main technical data.

### Visual-priority rule

- Do not use icon-only primary actions without a visible text label or accessible text fallback.
- Avoid decorative hero treatments, oversized banners, or large color fields that compete with query context and data density.

## Information Architecture

### Primary navigation

- `Leads` remains the ingestion and pipeline page.
- Add `Manufacturers` as the Phase 5 discovery page.
- Do not add Phase 6 workflow actions or notes surfaces here.

### Routes

| Route | Purpose |
|-------|---------|
| `/leads` | Existing lead import and lifecycle tracking |
| `/manufacturers` | Search and discovery dashboard |
| `/manufacturers/[leadId]` | Manufacturer detail view |

### Page structure

1. `/manufacturers` opens directly into the search dashboard, not a landing page.
2. `/manufacturers/[leadId]` is a dedicated route, not a modal, drawer, or tab.
3. Detail information must live on one continuous page with stacked sections; do not split technical data into tabs.

## List Page Contract

### Layout

- Use the existing dashboard shell and sidebar.
- Replace the narrow Phase 1 page width with a wide data layout: content width may expand to `max-w-[1600px]`.
- Keep page padding compact: `px-8 py-8` desktop, `px-4 py-6` mobile.
- The search/filter toolbar must sit above the results table and remain visible while scrolling the first viewport on desktop.

### Results summary

- Show a `Label`-sized results count directly under the toolbar.
- Format: `{N} manufacturers`.
- When filters are active, append `{M} filters applied`.

### Results table

Use a dense table on desktop and tablet. Switch to stacked result cards only below `768px`.

| Column | Width | Content contract |
|--------|-------|------------------|
| Manufacturer | 280px, sticky left | Company name link, domain or source URL beneath |
| Products | 260px | Up to 2 chemical names, then `+N more` |
| CAS | 160px | Up to 2 CAS numbers, stacked; exact matches highlighted |
| Industry | 180px | Up to 2 chips, then `+N` |
| Location | 180px | Primary city/state or `Location not extracted`; secondary `+N more` if needed |
| Capacity | 120px, right-aligned | Normalized `MT/year`, or raw text fallback |
| Status | 120px, centered | Semantic status badge |

### Row rules

- Default row height is visually dense but never shorter than 64px.
- Manufacturer name is the primary navigation affordance; do not make the entire table row clickable.
- Secondary metadata inside cells uses `Label` sizing and muted zinc text.
- Query highlights use accent tint only behind exact or strong matches; do not highlight entire cells.
- Rows without profile data stay visible but muted, with em dashes or explicit empty copy instead of blank cells.

### Default ordering

- If a search query is present, order by search relevance first, then manufacturer name A-Z.
- If no search query is present, order by status priority `Extracted` -> `Crawled` -> `Processing` -> `New` -> `Errored`, then manufacturer name A-Z.

### Pagination

- Use paginated results once the match set exceeds 50 rows.
- Footer controls are compact and right-aligned.
- Show current range format: `1-50 of 238`.

### Mobile rendering

- Below `768px`, replace the wide table with stacked result cards.
- Each card shows: manufacturer, top products, top CAS, location, capacity, and status.
- Filters move into a sheet; search stays inline at the top.

## Filters And Search Contract

### TanStack Table patterns

Use TanStack Table concepts for the interaction model:
- global search state
- faceted filters
- sorting state
- pagination state
- column visibility state internally only if needed

Do not expose row selection, bulk actions, inline editing, or drag-reordering in Phase 5.

### Search behavior

- Search must match across manufacturer name, lead URL/domain, product name, CAS number, industry text, city, state, and country.
- If the query matches CAS formatting, exact CAS matches rank above text matches.
- Search input updates results after a 250ms debounce on desktop and on explicit submit on mobile.
- Pressing `Enter` in the search field triggers the same search action as the primary CTA.

### Filter set

| Filter | Control | Behavior |
|--------|---------|----------|
| Industry | Multi-select command popover | Match any selected industry |
| Capacity | Single-select bucket menu | `Unknown`, `<100`, `100-999`, `1,000-9,999`, `10,000+ MT/year` |
| Status | Multi-select popover | Allow mixed lifecycle states |
| Location | Multi-select command popover | Filter by state first, city display remains informational |

### Applied filter presentation

- Show active filters as inline chips below the toolbar on desktop.
- Each chip must be individually removable.
- `Clear filters` resets all filter state but preserves the current search query.

### URL state

- Persist search and filter state in the URL query string.
- At minimum preserve: `q`, `industry`, `capacity`, `status`, `location`, `page`.
- Opening a detail page and returning to the list must restore the prior list state.

## Detail Page Contract

### Layout

- Use a two-column desktop layout: main content plus a sticky right rail.
- Suggested desktop grid: `minmax(0,1fr) 320px`.
- Collapse to a single column below `1024px`.
- Keep every data section visible on one route; do not use tabs.

### Header summary

The top of the page must show:
- manufacturer name
- website/domain
- pipeline status badge
- extraction freshness (`Extracted {date}`)
- back link to `Manufacturers`

If capacity is available, surface it in the header summary. If not, do not reserve an empty metric slot.

### Main column sections

#### 1. Products & CAS

- Render as a compact table.
- Columns: `Chemical`, `CAS`, `Grade`.
- Default order is alphabetical by product name.
- If `casNumber` is missing, show `No CAS`.
- If `grade` is missing, show an em dash.

#### 2. Locations

- Render each location as a bordered list row, not a map.
- Primary line: city, state.
- Secondary line: full address if available.
- Country is shown only when not `India`.

#### 3. Source Coverage

- Render crawl source URLs grouped by `homepage`, `products`, `about`, `other`.
- Show plain page-type labels and source links.
- This is read-only provenance, not a content viewer.

### Sticky right rail

#### 1. Snapshot

- Lead status
- Lead source URL
- Last extracted timestamp
- Counts for products, contacts, and locations

#### 2. Industries Served

- Show as compact chips or stacked pills.
- If empty, show `No industries extracted`.

#### 3. Capacity

- Preferred format: normalized value with thousands separators, followed by `MT/year`.
- If normalized value is missing but `capacityRawText` exists, show raw text under the label `Raw extracted capacity`.
- If both are missing, show `Capacity not extracted`.

#### 4. Contacts

- Order contact groups as `Email`, `Phone`, `WhatsApp`.
- Render values as actionable links when format allows.
- If multiple values exist for a type, stack them with 8px gaps.

### Sparse-data rule

- Never hide a section solely because its data array is empty.
- Each empty section must render its heading plus a one-line empty message.
- If the lead exists but `manufacturer_profiles` has no row, render the detail page in a profile-empty state using lead metadata only.

## State Handling

| Context | Contract |
|---------|----------|
| List loading | Show search toolbar skeleton plus 8 table-row skeletons. Preserve layout width to avoid reflow. |
| List empty dataset | Full-width empty state card with `No manufacturers available` copy. |
| List no match | Replace result rows with no-match state and keep the toolbar visible. |
| List partial data | Show explicit placeholders such as `No CAS`, `Location not extracted`, or em dash. |
| List error | Inline error surface above the table; preserve the current query and filter controls. |
| Detail loading | Skeleton header, one product-table skeleton, and two stacked section skeletons. |
| Detail profile-empty | Show lead metadata, status, and the `No extracted profile available` callout. |
| Detail partial data | Keep all sections visible with empty-state copy per section. |
| Detail not found | Centered not-found state: `Manufacturer not found`. Secondary copy: `Return to manufacturers and choose another profile.` |
| Detail error | Full-width error card with `Back to manufacturers` link. |

## Accessibility Contract

- Search input must have a programmatic label. Placeholder text does not replace the label.
- Sortable table headers must expose `aria-sort`.
- Result count changes after search/filter updates must be announced via a polite live region.
- Status meaning must never rely on color alone; text labels remain visible in every badge.
- Manufacturer links, filter chips, popover options, and pagination controls must all be keyboard reachable.
- Applied filter chips must expose clear remove labels such as `Remove industry: Pharma`.
- Mobile filter sheet must trap focus and restore focus to the trigger on close.
- Capacity values should expose a readable label such as `12,500 metric tons per year`.
- Empty and error states must remain inside the normal page reading order; do not rely on toast-only messaging.
- Maintain visible focus styles on every interactive control using the existing ring pattern.
