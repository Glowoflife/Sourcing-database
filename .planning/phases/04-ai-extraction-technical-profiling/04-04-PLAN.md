**File 2: CREATE src/extraction/run.ts**

Mirror src/acquisition/run.ts exactly, with these substitutions. The dotenv import MUST be the very first line (this is a hard constraint from Phase 3 — see RESEARCH.md Pitfall 6 and STATE.md session record).

```typescript
// MUST be the very first import — loads DATABASE_URL and REDIS_URL before singletons initialize
import { config } from "dotenv";
config({ path: ".env.local" });

// All other imports AFTER dotenv
import { db } from "@/db/index";
import { leads } from "@/db/schema";
import { extractionQueue } from "@/workers/queues";
import { logger } from "@/lib/logger";
import { eq } from "drizzle-orm";

async function main(): Promise<void> {
  // Query all leads with status = "Crawled" — these are ready for AI extraction
  // (No stale-reconciliation needed: extraction does not use an interim Processing state)
  const crawledLeads = await db
    .select({ id: leads.id })
    .from(leads)
    .where(eq(leads.status, "Crawled"));

  if (crawledLeads.length === 0) {
    logger.info({ stage: "extract", status: "ok", message: "No Crawled leads to enqueue" });
    await extractionQueue.close();
    return;
  }

  // addBulk uses a Redis pipeline — commands are batched for efficiency but NOT atomic.
  await extractionQueue.addBulk(
    crawledLeads.map((lead) => ({
      name: "extract",
      data: { leadId: lead.id },
    })),
  );

  logger.info({
    stage: "extract",
    status: "ok",
    message: `Enqueued ${crawledLeads.length} extraction jobs`,
  });

  // REQUIRED: close the Redis connection so the process exits cleanly
  await extractionQueue.close();
}

main().catch((err) => {
  logger.error({ stage: "extract", status: "fail", message: `Fatal: ${String(err)}` });
  process.exit(1);
});
```

Key differences from acquisition/run.ts:
- Queries `status = "Crawled"` (not "New")
- No stale-reconciliation logic (extraction has no interim Processing state)
- Job data is `{ leadId: lead.id }` only — no `url` field needed (extraction reads pages from DB)
- No URL validation filter needed (extraction does not use URLs)
- Stage is "extract" throughout

| npm run extract CLI → BullMQ queue | CLI enqueues job payloads; leadId is a positive integer from DB — no user-supplied input at this layer |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-11 | Tampering | BullMQ job payload | mitigate | `ExtractionJobSchema.safeParse(job.data)` validates leadId is a positive integer before processing — invalid payloads throw immediately, logged as failed jobs |
| T-04-12 | Denial of Service | extractionQueue.addBulk | accept | addBulk is a Redis pipeline — not atomic. A partial failure leaves some jobs enqueued. Acceptable: re-running `npm run extract` is idempotent (leads already Extracted will not be re-queued because query filters `status = "Crawled"`) |
