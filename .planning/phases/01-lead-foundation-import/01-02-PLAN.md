# Plan 01-02: CSV Import API and Leads List API

<objective>
Implement the API surface that satisfies DISC-03 (CSV import) and exposes lead data for the dashboard. Two route handlers: `POST /api/leads/import` accepts a multipart `file` field, parses the CSV with `csv-parse/sync` (BOM-stripped), validates each row against a Zod `ImportRowSchema`, and bulk-inserts with `onConflictDoNothing` on the `url` unique constraint (the chosen duplicate-handling strategy under "Claude's Discretion" in CONTEXT.md). `GET /api/leads` returns all leads ordered by `createdAt DESC` for the Plan 01-03 dashboard. A sample CSV fixture is committed so the import flow can be exercised end-to-end without external data.
</objective>

<threat_model>
## Threat Model (ASVS L1)

| Trust Boundary | Description |
|----------------|-------------|
| browser → /api/leads/import | Untrusted multipart body crosses into Node.js — file content, filename, content-type all attacker-controlled |
| browser → /api/leads | Untrusted GET request — currently no auth boundary in Phase 1 (internal tool, single-user dev) |

| Threat ID | Category (STRIDE) | Component | Disposition | Mitigation Plan |
|-----------|-------------------|-----------|-------------|-----------------|
| T-02-01 | Tampering / Injection | CSV row → SQL INSERT | mitigate | Drizzle parameterised queries — never string-interpolate row values into SQL. `db.insert(leads).values(validRows)` uses bind params; SQL injection via CSV data is not possible |
| T-02-02 | Denial of Service | Oversized CSV upload | mitigate | Reject files larger than 10 MB before parsing (`file.size > 10 * 1024 * 1024` returns 413). Per RESEARCH.md A3, sync parsing is acceptable below this size |
| T-02-03 | Tampering | Malicious file masquerading as CSV | mitigate | Validate `file.name.endsWith('.csv')` AND attempt `csv-parse` — non-CSV content fails parsing; on parse failure return 422 with sanitized error (no raw bytes echoed) |
| T-02-04 | Information Disclosure | Stack trace leakage in error responses | mitigate | Catch parse errors and Zod errors; return only a structured `{error, errors[]}` body. Never include `error.stack` or full request body in the response |
| T-02-05 | Tampering / Path Traversal | Filename used to construct a path | accept (mitigated by design) | Filename is read from `formData` but is NEVER used to construct a filesystem path; the buffer is read from `file.arrayBuffer()` directly. No path traversal vector exists |
| T-02-06 | Repudiation | Loss of import audit trail | mitigate | `logger.info({stage:"import", status:"ok", durationMs, errorCount})` after every import; `logger.warn` for rejected rows. CONVENTIONS.md structured logging requirement |

</threat_model>

<action>
Create `src/app/api/leads/import/route.ts`. This is based on RESEARCH.md Pattern 4, extended with the threat-model mitigations (size cap T-02-02, file extension check T-02-03, structured response without stack traces T-02-04, structured logging T-02-06):

```typescript
import { parse } from "csv-parse/sync";
import { db } from "@/db";
import { leads } from "@/db/schema";
import { ImportRowSchema } from "@/schemas/import.schema";
import { logger } from "@/lib/logger";

const MAX_FILE_BYTES = 10 * 1024 * 1024; // 10 MB

export async function POST(request: Request) {
  const startedAt = Date.now();
  let formData: FormData;

  try {
    formData = await request.formData();
  } catch {
    logger.warn({ stage: "import", status: "fail", message: "invalid multipart body" });
    return Response.json({ error: "Invalid multipart form data" }, { status: 400 });
  }

  const file = formData.get("file");
  if (!(file instanceof File)) {
    logger.warn({ stage: "import", status: "fail", message: "no file field" });
    return Response.json({ error: "No file uploaded. Use form field 'file'." }, { status: 400 });
  }

  // T-02-02: reject oversized uploads before parsing
  if (file.size > MAX_FILE_BYTES) {
    logger.warn({ stage: "import", status: "fail", message: `file too large: ${file.size}` });
    return Response.json(
      { error: "File exceeds 10 MB limit." },
      { status: 413 },
    );
  }

  // T-02-03: extension check (defense in depth — parsing also rejects non-CSV)
  if (!file.name.toLowerCase().endsWith(".csv")) {
    logger.warn({ stage: "import", status: "fail", message: `non-csv extension: ${file.name}` });
    return Response.json(
      { error: "File must have .csv extension." },
      { status: 415 },
    );
  }

  const buffer = Buffer.from(await file.arrayBuffer());
  let rows: unknown[];

  try {
    rows = parse(buffer, {
      columns: true,
      skip_empty_lines: true,
      trim: true,
      bom: true, // RESEARCH.md Pitfall 2 — strip UTF-8 BOM from Excel exports
    });
  } catch {
    // T-02-04: do NOT echo raw error or buffer
    logger.warn({ stage: "import", status: "fail", message: "csv parse failed" });
    return Response.json(
      { error: "Could not read the file. Make sure it is a valid CSV with name and url columns." },
      { status: 422 },
    );
  }

  const validRows: { name: string; url: string }[] = [];
  const errors: { row: number; error: string }[] = [];

  rows.forEach((row, i) => {
    const result = ImportRowSchema.safeParse(row);
    if (result.success) {
      validRows.push(result.data);
    } else {
      // +2 because: 1-indexed display + 1 header line
      errors.push({ row: i + 2, error: result.error.issues.map((iss) => iss.message).join("; ") });
    }
  });

  let inserted = 0;
  if (validRows.length > 0) {
    const result = await db
      .insert(leads)
      .values(validRows)
      .onConflictDoNothing({ target: leads.url })
      .returning({ id: leads.id });
    inserted = result.length;
  }

  const skipped = validRows.length - inserted;
  const durationMs = Date.now() - startedAt;

  logger.info({
    stage: "import",
    status: "ok",
    durationMs,
    errorCount: errors.length,
    message: `inserted=${inserted} skipped=${skipped} totalRows=${rows.length}`,
  });

  return Response.json({ inserted, skipped, errors });
}
```

Notes on the implementation:
- The response shape `{inserted, skipped, errors}` matches what UI-SPEC's `ImportResultFeedback` component renders.
- `onConflictDoNothing({target: leads.url})` is the Phase 1 duplicate-handling strategy (Claude's Discretion per CONTEXT.md).
- `bom: true` is non-negotiable per RESEARCH.md Pitfall 2.
- Zod error mapping uses `result.error.issues` (Zod 4 API) — the `.message` from `result.error.message` would return a JSON-encoded string in Zod 4, which is verbose. Joining `issues[].message` is more user-friendly.
</action>

```typescript
import { db } from "@/db";
import { leads } from "@/db/schema";
import { desc } from "drizzle-orm";

export async function GET() {
  const rows = await db
    .select()
    .from(leads)
    .orderBy(desc(leads.createdAt));

  return Response.json(rows);
}
```

Notes:
- No auth boundary in Phase 1 — internal tool, no exposed deployment yet. Adding auth is a v2 / later-phase concern.
- The response is the raw `Lead[]` array; the dashboard Server Component (Plan 01-03) calls `db` directly for SSR but the endpoint exists for client-side refresh after import (`router.refresh()` triggers re-render of the Server Component, which re-runs `db.select()`, so this endpoint is largely a future hook — committing it now keeps the API surface complete and avoids a follow-up plan).
- The `desc` import comes from `drizzle-orm`, NOT `drizzle-orm/pg-core`.
</action>
